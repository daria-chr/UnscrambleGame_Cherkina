# UnscrambleGame

## Описание

Игра на угадывание слов. Игроку показывается перемешанное слово, и он должен ввести правильный вариант. За каждый правильный ответ начисляются очки. Игра состоит из 10 раундов, после чего показывается итоговый счёт.

## Технологии

- Kotlin
- Jetpack Compose
- ViewModel
- StateFlow

## Функциональность

- Отображение перемешанного слова, которое нужно угадать
- Ввод ответа с проверкой правильности
- Начисление очков за каждый правильный ответ
- Отображение ошибки при неправильном ответе
- Возможность пропустить слово без изменения счёта
- Счётчик текущего вопроса
- Сохранение состояния игры при повороте экрана благодаря

## Контрольные вопросы

### 1. Что такое ViewModel и зачем он нужен?

ViewModel — это специальный класс из Android Jetpack, который хранит данные и логику UI отдельно от Composable-функций и Activity. Он решает проблему потери состояния при изменении конфигурации устройства (например, при повороте экрана). Без ViewModel при повороте экрана Activity уничтожается и создаётся заново, а все обычные переменные сбрасываются в начальные значения. ViewModel же остаётся в памяти и переподключается к новой Activity, сохраняя все данные.

### 2. В чём разница между MutableStateFlow и StateFlow?

**MutableStateFlow** — изменяемый поток, значение которого можно менять.
**StateFlow** — неизменяемое представление того же потока, доступное только для чтения.

### 3. Почему в ViewModel нельзя хранить Context или View?

ViewModel живёт дольше, чем Activity. Если сохранить ссылку на Activity внутри ViewModel, то при пересоздании Activity старый объект не будет уничтожен сборщиком мусора, так как ViewModel всё ещё держит на него ссылку. Это приведёт к утечке памяти.

### 4. Чем StateFlow отличается от remember и rememberSaveable?
- StateFlow — применяется для хранения состояния на уровне ViewModel и обеспечивает реактивность между ViewModel и пользовательским интерфейсом.
- remember — используется внутри composable-функций для хранения состояния, которое сохраняется при рекомпозиции, но теряется при изменении конфигурации.
- rememberSaveable — работает аналогично remember, но сохраняет состояние при изменении конфигурации (например, при повороте экрана).

**Когда использовать?**
- StateFlow — для глобального состояния приложения или экрана.
- remember — для временного состояния внутри одного composable.
- rememberSaveable — когда требуется сохранять состояние интерфейса при повороте экрана.

### 5. Что такое data class в Kotlin и зачем он используется для UI State?

`data class` — специальный тип класса в Kotlin, для которого компилятор автоматически генерирует методы:
- `equals()` — сравнение объектов по значению полей
- `hashCode()` — хэш на основе полей
- `toString()` — читаемое строковое представление
- `copy()` — создание копии объекта с изменением отдельных полей

### 6. Как работает collectAsState() в Compose?

`collectAsState()` подписывает Composable-функцию на StateFlow. Каждый раз, когда значение StateFlow меняется, Compose автоматически перезапускает все функции, которые используют это значение. Таким образом UI всегда отображает актуальное состояние без ручного вызова обновлений.

### 7. Зачем разделять код на пакеты data и ui?

Разделение на пакеты поддерживает принципы разделения ответственности и чистой архитектуры:
- Пакет `data` содержит модели данных и константы — то, что не зависит от UI
- Пакет `ui_model` содержит ViewModel и Composable-функции — логику отображения

Это делает код более читаемым, удобным для тестирования и позволяет нескольким разработчикам работать над разными частями проекта независимо.

### 8. Что происходит с ViewModel, когда приложение полностью закрывается (process death)?

При полном закрытии приложения ViewModel не сохраняется, все данные теряются. ViewModel решает только проблему изменения конфигурации (поворот экрана, смена темы).

Для долговременного хранения данных используются другие подходы:
- **Room Database** — локальная база данных на устройстве
- **DataStore** — хранение простых данных (замена SharedPreferences)
- **SavedStateHandle** внутри ViewModel — сохраняет небольшой объём данных в Bundle, переживает process death
